## 0. 現況盤點（可直接沿用的地基）

| 能力 | 現況 | 位置 |
|---|---|---|
| 帳號 | Google 登入 → 簽發 App JWT，使用者以 **email** 為主鍵 | `lib/auth.js`、`lib/users.js` |
| JWT 驗證 | `verifyAppJwt(token)`、`bearerFromReq(req)` 已可從 `Authorization: Bearer` 取 token | `lib/auth.js` |

## 1. 缺的中介層：`requireAuth`

所有社群/好友/共享 API 都要先確認「你是誰」。新增一個 middleware：

```js
// lib/requireAuth.js
const { verifyAppJwt, bearerFromReq } = require('./auth');
module.exports = function requireAuth(req, res, next) {
  const token = bearerFromReq(req);
  const payload = token && verifyAppJwt(token);
  if (!payload || !payload.email) {
    return res.status(401).json({ error: '請先登入' });
  }
  req.userEmail = payload.email;   // 之後所有 handler 用這個當「我是誰」
  next();
};
```

前端所有社群請求都要帶 `Authorization: Bearer <token>`（token 已存在 `AUTH_LS`）。

## 2. 資料模型

以 email 當使用者主鍵；為了對外顯示與加好友，另給每個帳號一組不可猜的 **friendCode**（例如 `PET-8F3K2Q`）。

### 2.1 使用者公開檔（profile）
```
User {
  email        // 主鍵（內部）
  friendCode   // 對外加好友用，唯一、隨機
  displayName  // 暱稱（預設用寵物名）
  petAnimal    // 生肖 → 頭像
  createdAt
}
```

### 2.2 好友關係（friendship）
用「一列代表一段雙向關係」＋狀態機：
```
Friendship {
  id
  requester     // email，送出邀請者
  addressee     // email，收到邀請者
  status        // 'pending' | 'accepted' | 'blocked'
  createdAt
  respondedAt
}
```
- 查「我的好友」= status='accepted' 且 (requester=我 或 addressee=我)。
- 查「待我回應的邀請」= status='pending' 且 addressee=我。

### 2.3 動態貼文（post）／按讚／留言
```
Post {
  id
  authorEmail
  kind          // 'user' | 'achievement' | 'system'
  text
  meta          // 成就貼文的結構化資料（存錢率、連續天數、聖戰戰績…）
  createdAt
}
Like    { postId, userEmail, createdAt }          // (postId,userEmail) 唯一
Comment { id, postId, authorEmail, text, createdAt }
```
**動態可見範圍（v1 建議）**：好友 + 自己。查詢「我的動態牆」= 作者 ∈ (我的好友 ∪ 我) 的貼文，時間排序。

### 2.4 好友版分帳（把現有 group 綁到帳號）
在現有 split-bill group 上加兩個東西：
```
group.ownerEmail            // 建立者
group.memberLinks: {         // 把「分帳成員」對應到「真實帳號」（可選）
  [memberId]: email | null   // null = 純掛名、非 App 使用者
}
group.sharedWith: [email]    // 有權存取此 group 的帳號（= owner + 被邀請的好友）
```
- 存取控制：`/api/split-bill/groups/:id/*` 一律先過 `requireAuth`，再檢查 `req.userEmail ∈ group.sharedWith`，否則 403。
- 邀請好友加入分帳 = 把好友 email push 進 `sharedWith`（且對方要是你的好友）。
- 成員綁帳號後，該成員在「結算」看到的「我應收/應付」就會對到自己。

## 3. API 一覽（皆需 `requireAuth`）

### 好友 `/api/friends`
| Method | Path | 說明 |
|---|---|---|
| GET | `/me` | 取得我的 profile + friendCode |
| GET | `/` | 我的好友清單 |
| GET | `/requests` | 待我回應的邀請 |
| POST | `/requests` | 送出邀請 `{ friendCode }` 或 `{ email }` |
| POST | `/requests/:id/accept` | 接受邀請 |
| POST | `/requests/:id/decline` | 拒絕 |
| DELETE | `/:email` | 移除好友 |

### 動態 `/api/feed`
| Method | Path | 說明 |
|---|---|---|
| GET | `/` | 我的動態牆（好友＋自己，分頁 `?before=cursor`） |
| POST | `/posts` | 發文 `{ text, kind, meta }` |
| DELETE | `/posts/:id` | 刪自己的貼文 |
| POST | `/posts/:id/like` | 按讚 / 取消讚（toggle） |
| POST | `/posts/:id/comments` | 留言 `{ text }` |

### 分帳（在既有路由上加權限與共享）
| Method | Path | 變更 |
|---|---|---|
| （全部） | `/api/split-bill/**` | 前面掛 `requireAuth` + `sharedWith` 檢查 |
| POST | `/groups/:id/share` | 邀好友進此分帳 `{ email }`（限好友） |
| POST | `/groups/:id/members/:mid/link` | 把成員綁到帳號 `{ email }` |
| GET | `/groups` | 只回「我有權存取」的 group |

## 5. 持久化與遷移

社群/好友**不建議用 JSON 檔**（多人同時寫、關聯查詢會爆）。建議直接上 Postgres（你已有 `lib/db-pg.js`）。

### 5.1 建議 schema（Postgres）
```sql
CREATE TABLE users (
  email TEXT PRIMARY KEY,
  friend_code TEXT UNIQUE NOT NULL,
  display_name TEXT,
  pet_animal TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
CREATE TABLE friendships (
  id BIGSERIAL PRIMARY KEY,
  requester TEXT REFERENCES users(email),
  addressee TEXT REFERENCES users(email),
  status TEXT NOT NULL DEFAULT 'pending',
  created_at TIMESTAMPTZ DEFAULT now(),
  responded_at TIMESTAMPTZ,
  UNIQUE (requester, addressee)
);
CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,
  author_email TEXT REFERENCES users(email),
  kind TEXT NOT NULL DEFAULT 'user',
  text TEXT,
  meta JSONB,
  created_at TIMESTAMPTZ DEFAULT now()
);
CREATE TABLE post_likes (
  post_id BIGINT REFERENCES posts(id) ON DELETE CASCADE,
  user_email TEXT REFERENCES users(email),
  created_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (post_id, user_email)
);
CREATE TABLE post_comments (
  id BIGSERIAL PRIMARY KEY,
  post_id BIGINT REFERENCES posts(id) ON DELETE CASCADE,
  author_email TEXT REFERENCES users(email),
  text TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
-- 分帳共享：沿用現有 group JSON，另加索引表方便「查我能看的 group」
CREATE TABLE split_group_access (
  group_id TEXT,
  user_email TEXT REFERENCES users(email),
  PRIMARY KEY (group_id, user_email)
);
```

### 5.2 Render 免費方案注意
- 免費 Postgres 會過期／被停用；免費 web service 檔案系統是暫時的。做「線上社群」前，建議把 DB 換成**不會過期**的方案（Render 付費 DB、Neon、Supabase 等），否則好友/動態資料會週期性消失。
- 這也是先前「部署失敗」的同源風險：`server.js` 在 `DATABASE_URL` 連不上時會 `process.exit(1)`。上線前務必確認 DB 穩定。

## 6. 隱私與安全（不能省）
- 所有社群 API 一律 `requireAuth`；資源存取一律驗「你是不是這筆資料的關係人」（好友、貼文作者、group 成員）。
- friendCode 用隨機不可猜字串，不要用 email 直接曝露。
- 動態預設只給好友看；未來若要「公開貼文」再加 `visibility` 欄位。
- 分帳金額涉及個資，共享務必先驗好友關係 + `sharedWith`。

---

## 7. 建議落地順序
1. **`requireAuth` + users/profile + friendCode**（最小可用帳號公開檔）。
2. **好友系統**（邀請碼加好友、清單、接受/拒絕）。
3. **動態牆上線**（把前端 `S.social` 換成 `/api/feed`；加好友可見範圍）。
4. **成就自動發文**（接遊戲事件）。
5. **好友版分帳**（split-bill 加 `requireAuth` + `sharedWith` + 成員綁帳號 + 邀好友）。

> 前端第 1、3、5 的接點都已在這次改動中鋪好，屆時多半只是把「改 `S.xxx`」換成「呼叫 API」。
